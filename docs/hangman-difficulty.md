# Hangman word difficulty

Hangman's "easy/medium/hard/nightmare" word pools (`games/words/hangman_<bin>.txt`) aren't a manual
curation or a simple frequency split — they're derived by actually **simulating a hangman solver**
against every candidate word and scoring how hard each one was to guess, then filtering by how
common the word is. This doc records the methodology, the corpus, and the exact thresholds chosen,
so the choices can be revisited without re-deriving them from scratch.

Generated by `scripts/compute_hangman_difficulty.py`; full per-word output saved to
`data/hangman_word_difficulty.csv`.

## Corpus

Source: `data/CSW21.txt`, Collins Scrabble Words 2021 (a definition-format word list — the word is
the first whitespace-separated token per line). Filtered to alphabetic words **3–15 letters**
(`MIN_LENGTH`/`MAX_LENGTH` in the script) — the same source already used to generate Wordle's word
lists, and the only change needed to support hangman words much longer than Wordle's 7-letter cap.

This yields **278,950 candidate words**, grouped by length for the solver (length-9 is the largest
bucket at 42,890 words; length 3 the smallest at 1,340).

## Difficulty: simulating a frequency-based hangman solver

### The solver

For each word, a deterministic solver plays it out from scratch (no 6-guess cap — this measures the
word's *inherent* difficulty, independent of the live game's guess limit):

1. Maintain the set of candidate words (same length) still consistent with everything revealed or
   ruled out so far.
2. Among letters not yet guessed, guess whichever one appears in the **largest fraction `p`** of
   remaining candidates (ties broken by overall corpus letter frequency, then alphabetically, for
   determinism).
3. If the letter is actually in the secret word: narrow the candidate set to words matching its
   *exact* revealed positions (real hangman reveals every occurrence of a guessed letter, not just
   "at least one").
4. If not: narrow the candidate set to words that don't contain that letter at all.
5. Repeat until every letter of the word has been guessed.

This is the standard, well-documented practical approach to hangman-solving — see
[dCode's Hangman solver](https://www.dcode.fr/hangman-game-solver) ("calculates, for each possible
word in the dictionary, the most likely letters based on the known positions... suggests the most
statistically relevant letter") and
[DataGenetics, "A Better Hangman Strategy"](http://datagenetics.com/blog/april12012/index.html)
(letter frequency should be computed *per word length*, not flat across English, since e.g. S
overtakes T as the second-most-common letter once you condition on word length).

Implementation note: candidate filtering and per-letter frequency counting are vectorized with
numpy (a `(N, length)` character-code array + an `(N, 26)` boolean presence matrix per length
bucket, built once and reused for every word of that length) — a naive per-word/per-step Python
loop over tens of thousands of candidates would be far too slow at this corpus size. The full run
takes a few minutes.

### The score: surprisal (cross-entropy / log-loss), not raw wrong-guess count

At each step the solver's guess carries an implicit prediction: `p` = "I believe there's a `p`
chance this letter is present." The actual outcome is scored by **surprisal** — Shannon's
self-information, `−log2(P(observed outcome))` — using the solver's own predicted `p`:

- If the letter **is** present: cost = `−log2(p)`.
- If **not**: cost = `−log2(1 − p)`.

The word's `difficulty_score` is the sum of these costs across every step. This is the same
mathematical object as **binary cross-entropy / log-loss** (the standard loss function for scoring
probabilistic predictions in ML) and as the **logarithmic scoring rule** (Good, 1952) for evaluating
forecasts — not a metric invented for this script, an established way to score "how well did a
sequence of predictions match what actually happened."

Why this, and not just "number of wrong guesses"? Because it captures *how surprising* each wrong
guess was, not just that it happened: a wrong guess the solver was 90% confident about costs far
more (`−log2(0.1) ≈ 3.32` bits) than a wrong guess it was only 55% confident about
(`−log2(0.45) ≈ 1.15` bits), and a word that's solved via several confident, correct guesses costs
almost nothing. More guessing steps (longer or more ambiguous words) also accumulate more cost.
This is exactly "the more likely the incorrect choices were, and the more steps it took, the higher
the difficulty" — quantified rather than just counted.

Strictly, **surprisal isn't the same thing as entropy** — entropy is the *expected* surprisal,
averaged over a distribution before the outcome is known; what's scored here is the *realized*
surprisal of the one outcome that actually happened for that specific word. They're related exactly
the way cross-entropy relates to entropy: if you averaged this score over every word consistent
with a given game state (the same candidate pool the solver computed `p` from), the average would
converge to that step's binary entropy `H(p) = −p·log2(p) − (1−p)·log2(1−p)`. A word's score is
best read as "how far this word's actual letters deviated from what the candidate-pool statistics
would lead you to expect."

### Alternatives considered (not adopted)

- **Information-theoretic / entropy-maximizing solver** — instead of maximizing P(present), pick
  the letter that maximizes *expected* entropy reduction over the resulting candidate-set partition
  (similar to entropy-based Wordle solvers). See
  [dazkins.com, "Solving Hangman With Information Theory"](https://dazkins.com/blog/solving-hangman-with-information-theory).
  This can pick a less-common letter if it splits the candidate set more evenly, which sometimes
  reduces *expected total guesses* — but hangman's cost function (wrong guesses specifically, not
  total turns) is better served by directly maximizing P(present), so this wasn't adopted.
- **"Evil Hangman" adversarial framing** (Stanford CS106B/L's classic assignment; formalized in
  [Barbay, "The Computational Complexity of Evil Hangman"](https://arxiv.org/abs/2003.10000)) —
  score the worst-case number of wrong guesses an adversarial word-setter could force while staying
  consistent with the revealed pattern, i.e. score a word's *ambiguity class* rather than the word
  itself. More rigorous, but answers a different question ("how bad could this pattern family be
  made to look") and is more expensive to compute. Not adopted.
- **Raw wrong-guess count** — simplest possible metric, used only as a secondary/debug column
  (`num_wrong` in the CSV), not the primary score: too coarse (most words differ by only 1-3 wrong
  guesses) and doesn't distinguish a narrow miss from a wildly-wrong confident guess.

## Commonness

`commonness` = [`wordfreq`](https://github.com/rspeer/wordfreq)'s English **zipf frequency** — a
log-scale word-commonness score (`zipf = log10(frequency per billion words)`), roughly `0` for a
word wordfreq doesn't recognize at all (mostly Scrabble-only words with no real-world usage data)
up to `~7.7` for extremely common words like "the". This is the *same* metric already used to
generate Wordle's word lists (`scripts/generate_wordlists.py`) and hangman's original
frequency-only tiers, kept here unchanged so the two scores are directly comparable per word.

Distribution across the full corpus is heavily skewed: **174,663 of 278,950 words (62.6%) score
exactly 0** (wordfreq has no data for them at all — almost all of these are valid-but-obscure
Scrabble words). Only the top ~5% of the corpus scores ≥3.44, and the top ~1% scores ≥4.49.

`wordfreq` is a generation-time-only dependency (installed via `uv run --with wordfreq`, never a
runtime dependency of the app), matching the convention already used for Wordle's lists.

## Binning: difficulty first, then a one-sided commonness floor per tier

Binning is **not** a single percentile split on `difficulty_score` — commonness also gates
eligibility, one-sidedly: easier tiers must be common, harder tiers tolerate progressively rarer
words, and `nightmare` has no floor at all.

1. Compute `difficulty_score` percentile cutoffs across the whole corpus: **50th / 80th / 95th**.
   A word's score below the 50th percentile is "easy by difficulty," below the 80th is "medium,"
   below the 95th is "hard," and above that is "nightmare" — purely by solver difficulty, before
   any commonness check.
2. Apply a **commonness floor per tier** (`COMMONNESS_FLOOR` in the script), demoting a word into
   the next harder tier if its commonness doesn't clear the floor for the tier its difficulty alone
   would put it in:

   | Tier | Commonness floor (zipf) | Meaning |
   |---|---|---|
   | `easy` | `≥ 4.0` | must be a genuinely common, everyday word |
   | `medium` | `≥ 2.5` | still recognizable |
   | `hard` | `≥ 1.0` | only mildly common acceptable |
   | `nightmare` | none | any commonness — only difficulty matters here |

   A word never lands in `easy`/`medium`/`hard` unless it clears that tier's floor; it's pushed into
   the next harder tier (cascading as far as `nightmare`, which always accepts) until one is
   satisfied. A word's bin can therefore only ever get *harder* than its raw difficulty percentile
   suggested, never easier.

The `4.0` floor matches the original (now-superseded) frequency-only "easy" threshold
(`scripts/generate_hangman_words.py`, deleted) for continuity — it's where the corpus naturally has
~6,450 commonly-recognized words. `2.5`/`1.0` were chosen by inspecting the commonness percentile
table above (roughly the 80th/63rd percentiles) so each looser tier meaningfully widens the eligible
pool without being toothless. These are tunable: re-run with `--rebin-only` (reads the existing CSV,
skips the ~5-8 minute solver simulation, just re-bins and rewrites
`games/words/hangman_<bin>.txt`) to try different cutoffs cheaply.

### Why this matters in practice

The two scores often disagree, which is the whole point of keeping both columns. Examples from the
generated table:

| Word | Commonness | Difficulty score | Bin | Why |
|---|---|---|---|---|
| `cat` | 4.78 (very common) | 8.67 (hard) | `hard` | common, but only 3 letters give the solver almost nothing to narrow down on |
| `the` | 7.73 (extremely common) | 7.63 (hard) | `hard` | same story — short common words are *not* automatically easy to guess |
| `absolute` | 4.54 | 3.30 (easy-range) | `easy` | common **and** structurally easy — exactly what "easy" should mean |
| `antipersonnel` | 1.71 (rare) | 0.39 (very easy-range) | `hard` | structurally trivial once a few letters land (long, distinctive), but too rare to call "easy" |
| `ecesis` | 0.0 (unrecognized) | 3.67 (easy-range) | `nightmare` | easy by difficulty alone, but completely obscure — correctly bottoms out at the no-floor tier |
| `kiwi` | 3.33 | 6.26 (medium-range) | `medium` | not quite common enough for `easy`'s 4.0 floor, demoted one tier |

### Resulting bin sizes

| Bin | Words | % of corpus |
|---|---|---|
| `easy` | 1,542 | 0.55% |
| `medium` | 22,847 | 8.19% |
| `hard` | 71,327 | 25.57% |
| `nightmare` | 183,234 | 65.69% |

`easy` is intentionally small — the *intersection* of "common" and "solver-easy" is narrow, since
(per the table above) many of the most common English words are short and don't disambiguate well.

## Output files

- `data/hangman_word_difficulty.csv` — one row per candidate word: `word, length, commonness,
  difficulty_score, num_steps, num_wrong, difficulty_bin`.
- `games/words/hangman_{easy,medium,hard,nightmare}.txt` — one word per line, what
  `games.wordlists.hangman_words(difficulty)` actually loads at runtime.

## Regenerating

```bash
# Full run (re-simulates the solver for every word, ~5-8 minutes):
uv run --with numpy --with wordfreq scripts/compute_hangman_difficulty.py --input data/CSW21.txt

# Re-bin only (reads the existing CSV, skips the solver — for tuning thresholds quickly):
uv run --with numpy --with wordfreq scripts/compute_hangman_difficulty.py --rebin-only
```
